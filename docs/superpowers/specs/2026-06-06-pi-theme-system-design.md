# Pi Theme System Design

## Goal

Make Pi inherit the dotfiles theme system from `.chezmoidata/themes.yml` without relying on `ghostty-sync` ANSI slot guesses. The result should stay dark, minimal, and subdued across every configured theme, with color used sparingly for meaningful state.

## Current context

- `.chezmoidata/defaults.yml` selects the active theme with `theme`.
- `.chezmoidata/themes.yml` defines each theme's semantic palette and per-app theme names.
- Ghostty already consumes the theme registry via `dot_config/ghostty/config.tmpl`.
- Pi currently uses a `ghostty-sync-*` theme generated by `pi-ghostty-theme-sync`; that extension maps Ghostty `palette[5]` to Pi `accent`, which produces pink/purple-heavy UI in themes where that ANSI slot is magenta.
- `pi-powerline-footer-local` supports a separate `theme.json` override for statusline segment colors, including `separator` and `border`.

## Approach

Chezmoi becomes the source of truth for Pi theme files. Ghostty and Pi become sibling outputs of `.chezmoidata/themes.yml`; Pi no longer depends on Ghostty's exported ANSI palette for semantic colors.

This means:

1. Generate a Pi theme JSON from the active chezmoi theme.
2. Generate the powerline footer theme override from the same active palette.
3. Ensure Pi's active theme in `~/.pi/agent/settings.json` is the stable generated theme name `chezmoi`.
4. Disable or neutralize `pi-ghostty-theme-sync` so it cannot overwrite the active Pi theme on startup.

## Generated files

### Pi TUI theme

Create a chezmoi-managed template that renders to a stable Pi theme file under:

```text
~/.pi/agent/themes/chezmoi.json
```

The theme `name` should stay `chezmoi` for every active dotfiles theme. The currently selected dotfiles theme can be recorded in `vars.activeTheme` for debugging, but Pi should always select the stable `chezmoi` theme name. This avoids stale generated theme files and avoids changing Pi settings on every theme switch.

Mapping rules:

- `bg`, `fg`, `success`, `warn`, and `error` map directly.
- Pi `accent` should avoid loud magenta/purple defaults. Prefer `primary` or `primary_alt`, depending on the palette, with `accentAlt` reserved for secondary highlights.
- Borders should come from subdued neutral palette slots, not from colorful accent slots.
- `border` should be low contrast but visible.
- `borderAccent` should be only a small step brighter/more colorful than `border`.
- `borderMuted` should be darker than `border`.
- Text defaults should generally remain terminal-default (`""`) where Pi supports that.

### Powerline footer theme

Create or template the file consumed by `pi-powerline-footer-local`, currently:

```text
~/.pi/agent/extensions/powerline-footer/theme.json
```

Mapping rules:

- `separator`: darker than the prompt/editor border. Use the theme's `border` or a darker neutral if available.
- `border`: slightly brighter than `separator`, matching the prompt-window wrapping lines.
- `model`, `path`, `gitClean`, `gitDirty`, and context states should use subdued semantic colors from the active palette.
- `thinking` states should remain distinguishable but not neon.
- The previous-prompt footer should be very low contrast. If the extension only uses its fixed `sep` color for that line, update the local extension to resolve it from theme configuration instead of the hard-coded ANSI 244.

## Ghostty-sync behavior

`pi-ghostty-theme-sync` should not remain the active authority for Pi theme selection. Prefer disabling its extension entry while leaving package history intact, for example by converting the package entry to an object with `extensions: []` if Pi supports that shape for this package. If that does not prevent startup theme writes, remove the package from Pi settings.

Ghostty itself remains managed by the dotfiles theme system through `dot_config/ghostty/config.tmpl`.

## Data flow

```text
.chezmoidata/defaults.yml / local.yml
          │
          ▼
.chezmoidata/themes.yml active palette
          │
          ├── dot_config/ghostty/config.tmpl
          ├── Pi theme template → ~/.pi/agent/themes/chezmoi.json (name = chezmoi)
          ├── powerline theme template → ~/.pi/agent/extensions/powerline-footer/theme.json
          └── Pi settings patch/script → ~/.pi/agent/settings.json theme = chezmoi
```

## Error handling and safety

- Templates should fail visibly if a theme is missing required palette keys.
- All themes in `.chezmoidata/themes.yml` should render with the same template logic.
- Existing local Pi settings that are unrelated to theming should be preserved where possible.
- The generated Pi theme name should be stable (`chezmoi`) and avoid hashes so it is predictable and easy to inspect.
- Avoid changing Ghostty ANSI palettes solely to satisfy Pi.

## Testing

1. Render templates for every theme in `.chezmoidata/themes.yml` with `chezmoi execute-template` or equivalent.
2. Validate generated Pi theme JSON parses and includes all required Pi color tokens.
3. Validate generated powerline `theme.json` parses.
4. Run `python3 scripts/validate-config-docs.py` after updating durable Pi config documentation.
5. Manually switch between at least `kanagawa-dragon`, `tokyonight-night`, and `rosepine` and confirm:
   - Pi starts with `chezmoi` selected.
   - Ghostty still uses the matching Ghostty theme.
   - statusline separators are darker than the prompt border.
   - prompt wrapping lines are subdued but readable.
   - previous-prompt footer is low contrast but legible.

## Out of scope

- Redesigning the entire Pi UI layout.
- Changing Ghostty's own theme definitions beyond existing theme selection.
- Maintaining `ghostty-sync` as the source of truth for Pi theme generation.
